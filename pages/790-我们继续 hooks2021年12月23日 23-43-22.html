上一篇文章描述了很多手段，避免不必要的渲染，以及如何得到正确的渲染调度问题

但是，实际生产中这些描述还不够，我们需要知道两个特别重要的知识

  1. 如果我不这么做，会发生什么事？
  2. 如果我不优化节点，会损失些什么？

首先，一个特别奇怪的例子：

function Child(props: {onChange: (res:string)=>void}){
    return <input onChange={e=>{
        props.onChange(e.target.value)
    }}/>
}
function Compo(){
    // 直接写变量
    let a = ""
    return <div>
        <Child onChange={res=>{
            // 直接赋值变量
            a = res
        ]}/>
        <button onClick={()=>{
            console.log(a)  // 打印得到正确结果
        }}>log</button>
    </div>
}

我不用任何 hooks，将 onChange 的 参数 赋值，在 log 按钮点击的时候，却能够得到正确的结果

所以，useRef 就没用了吗？这种做法也可以么？

答案是绝对不行！

因为假设我们在 Compo 中新加入一个 

const [b, setB] = useState(0)
useEffect(()=>{
    setInterval(()=>{
        setB(res=>++res)
    },1000)
    // 就不取消监听了，担待一下
},[])

这是一段和之前代码语义上完全不相关的逻辑，对吧？

但是你会发现，log 按钮点击时，打印结果时有时无

为什么？

因为 React 中，类型为 Dispatch 的方法（useReducer 的 dispatch 和 useState 的 setState），都会导致当前函数重新运行

而此处的 a 是个变量，它会在函数运行时重新赋值为 “”，导致之前的状态丢失

因此，如果 log 按钮点击前发生了任何其他依赖的状态变化（包括 context，state，props），a 都会被重置

直接写变量 —— 时有时无

你可能知道，React 中任何逻辑都应该包裹在 hooks 中，但是接受的都是正面信息，而这个负面例子会让你更深刻体会不这么做的后果

如果是一个没有 useMemo 或者 useRef 的数据，会发生这种后果，那么函数呢？

<div onClick={()=>{
    console.log(a)
}}/>

// 等效于

const aCb = ()=>{
   console.log(a)
}
<div onClick={aCb}/>

能否拿到正确的 a？

可以！那么这么做就是对的么？函数可以直接在 FC 中声明么？

答案也是绝对不行！

因为我们上面是在 React 的合成事件中绑定的 aCb，React 合成事件有类型为 DispatchEvent 的过程，可以视为默认加上了依赖

当我们用 eventListener 时，结果就是 “”

function Test2() {
  const [a, setA] = useState("");
  const buttonRef = useRef<HTMLButtonElement | null>(null);
  const cb = () => {
    console.log(a);
  };
  useLayoutEffect(() => {
    if (!buttonRef.current) return;
    const currentButton = buttonRef.current;
    currentButton.addEventListener("click", cb);
    return () => {
      currentButton.removeEventListener("click", cb);
    };
  }, []);
  return (
    <>
      <input
        value={a}
        onChange={(e) => {
          setA(e.target.value);
        }}
      />
      <button ref={buttonRef}>show a</button>
    </>
  );
}

你说不对，应该把 cb 的依赖加入 useLayoutEffect，这样就能拿到正确的 a 了

但是每次 FC 重新运行时，cb 都是新值（函数，对象，数组不判等）

因此，加入一个非 useCallback，useState 依赖，等同于没有加入依赖

那么，组件任何状态更新，都会导致 effect 运行，时机不对

回调不用 useCallback - 永远是空 / 时机不对（合成事件除外）

我们看到，function 无法直接判等，当然对象和数组也不行

能够直接写对象么？

你肯定会说，我怎么会直接写对象？既然都知道了直接写变量，直接写函数不对，傻子才会直接写对象呢

是么？

function Test3(props: {}) {

    console.log("props change");
  }, [props]);
  return <div>pure compo</div>;

function Test3Wrapper() {

你这里就直接写对象了喔 ~

哪有？

我们不用 jsx，我们用 createElement 来看，你就更加清楚和明白了：

function Test3(props: {}) {
  useEffect(() => {
    console.log("props change");
  }, [props]);
  return <div>pure compo</div>;
}
function Test3Wrapper() {
  const [a, setA] = useState("");
   return createElement(
     "div",
     {},
     Test3({}),
     createElement("input", {
       value: a,
       onChange: (e) => {
         setA(e.target.value);
       }
     })
   );
}

看到了么？Test3({}) 这里这个 大大 的 空对象？

所以，你的 useEffect props 就会打印很多次，神奇么？

当然，props 设为 props?: {} ，然后 Test3()，effect 就只执行一次了

直接写对象 - JSX 只是 createElement 语法糖

不写 jsx，直接写 createElement，又出现了新的问题：

函数能直接调用么？

函数直接调用意味着每次运行 FC，函数都会被调用一遍，这个问题 React 已经有过解法 ——

用 useMemo 包裹并填入特定依赖

在 FC 中大家都很习惯如此做，但是当 JSX 被才开，大家发现组件调用，createElement 也是函数调用时？

怎么解？

难道真要出现：

return <div>
    {useMemo(()=><input value={value} onChange={e=> {
        setValue(e.target.value)
    }}/>,[value])}
</div>

这种代码么？

这里，你就需要从成本收益的角度考虑问题了

新的 createElement，意味着新的 FiberNode，意味着新的 diff 过程

但是，一个节点的 diff 过程到底是多大的消耗？了解的人可以很快回答 O(n)

大部分情况下这是一个可以被忽略的问题

但是，性能瓶颈往往存在于 浏览器 渲染，而并非 MVVM 框架的 VM 生成和比对

我们写一个例子：

/* 重新 diff  */
function Test4(props: { n: number; value: string }) {
  const { n, value } = props;
  return (
    <div

我们优化这部分内容，假设我们只会在 第 1001 层使用 value 数据，我们传入精确的依赖

const ValueContext = createContext("");

// 仅在 Test4Final 处依赖 value 刷新
function Test4Final() {
  const value = useContext(ValueContext);
  return <div>{value}</div>;
}
function Test4Optimized(props: { n: number }) {
  const { n } = props;
  return (
    <div>
      <p>当前是第 {n} 项</p>
      {n < 1000 ? (
        <Test4Optimized n={n + 1} />
      ) : n === 1000 ? (
        <Test4Final />
      ) : null}
    </div>
  );
}

function Test4Wrapper() {
  const [value, setValue] = useState("");
  return (
    <div>
      <input
        value={value}
        onChange={(e) => {
          setValue(e.target.value);
        }}
      />
      <ValueContext.Provider value={value}>
        {useMemo(
          () => (
            <Test4Optimized n={0} />
          ),
          []
        )}
      </ValueContext.Provider>
    </div>
  );
}

可以看到，用 useMemo 和 useContext，与未出现 回流 的情况，其实好不了太多，换句话说：

跳过 diff，并没有真正指数意义上的体验优化和性能优化，仅仅是部分提升而已

但是，它能够精确规避特殊场景下的回流情形 —— 你能够精确控制跨层级节点的渲染

所以，你需要进行权衡，从道理上来讲

既然 组件 和 自定义 Hooks 内部需要用 useMemo 包裹，那么 ——

每一个 createElement 的函数和其参数 也应该由 useMemo 包裹

即，确定每一处视图应该如何跟随依赖项进行变化

当然这是不可能的，每个 jsx 节点套 useMemo 简直就是疯子，那么你没有这么做，目的是为了工作量和可读性

你就要明白你需要怎么做才能避免一些特殊情况下的坑

比如新版本的 useTransition 的加入，也可以用于解决而此类问题

函数重新运行这个问题，并不一定导致严重的性能问题，但是，你也绝不可能按照最精确的调度方式来写代码

这就是尤大说的，大部分情况下，React 性能差点

不过，性能差点，你得到了更加主动可控的东西，你完全可以无视成本将 React 写到性能很强的地步，但是你也有决断用最快的效率写出性能不差的代码

同时，Vue 的 proxy 虽然有 不依赖框架，更加直观的优点，但是在调度一致性上，比如没有 batch 等，在涉及依赖注入上，比如 proxy 被打断的问题，还有 ref 的心智负担上，都有缺陷

Vue 本质和 React 并没有区别，只是一个将 lane 放在函数上，一个放在了 proxy 上而已，Vue 在小应用和小问题上基本没有任何阻碍，但是容易在大项目和大问题上抓瞎，React 反过来了而已
