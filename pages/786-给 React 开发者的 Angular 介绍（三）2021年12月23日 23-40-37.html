我们来重点关注一下，React 社区经常会出现的话题 —— 状态管理

先抛出一个结论 —— 在 Angular 中不存在状态管理需求

先别急着讨论为什么会这样，我们先来了解，状态管理工具存在的意义

我们先抛开纯函数，单向数据流之类的领域，单单关注一个简单的过程 —— 在组件中发起事务：

function SomeCompo(){
   const [result, setResult] = useState<SomeType | null>(null)
   const submit = useCallback(()=>{
       request().then(res=>{
           setResult(res)
       })
   },[])
   // 节点合成事件可以不用 useCallback， 但是默认加上，展示依赖
   return <button onClick={submit}/>
}

这么写会出现一个问题 —— 组件销毁的时候，需不需要执行 promise resolve？

换句话说，对于一个有生命周期的封装单位，当他析构的时候，需不需要终止其中的异步操作？

结论是不需要，原因在与事务处理时的原子性要求：

对数据改操作要全部执行，要么全部不执行

submit, request, then, 要么全部都动，要么全部都不动，已经泼出去的水必须泼出去

因为 useCallback 内部，任意地方都可能 setState，对不对？

你能想象产品给出，操作完成，请求返回之后，不给用户提示的需求么？

但是，你真在 React 内部写代码的时候，会发现，组建销毁之后，setResult 运行会报错：

虽然这个报错影响不影响使用，不过一旦你加上了 useEffect 用以监听数据：

function SomeCompo(){
   const [result, setResult] = useState<SomeType | null>(null)
   const submit = useCallback(()=>{
       request().then(res=>{
           setResult(res)
       })
   },[])
   useEffect(()=>{
      // 副作用操作
      // 组件销毁后将不会执行
      // setXxx()
   },[result])

   return <button onClick={submit}/>
}

useEffect 以 result 作为依赖，一旦有操作，便不会运行，所以很大可能会违背： 对数据改操作要全部执行，要么全部不执行 的原则，破坏事务原子性

怎么办呢？

其实很简单，我不让数据和操作与组件生命周期相关联不就行了么？

最简单的方式 —— 在所有组件生命周期开始之前，就将数据（状态）和相关操作初始化 —— 也即 Redux

至于纯函数变更之类的，是确保一致性和隔离性，针对 Redux 做的持久化是为了持久性（不过持久性对前端来说，不是那么重要）

当然，这只是从事务视角对业务采用 redux 进行的一种解释，还有其他视角

当然，针对特定领域（模块），只要保证数据和操作先于领域内组件初始化，也可以解决这个问题 —— useReducer + useContext

在 React Hooks 中，假设不使用以上方案，你还可以采用 callback 嵌套的方式，即事务中不写 useEffect，甚至不 setState，而是在事务完成之后，才进行完整变更：

function SomeCompo(){
   const [result, setResult] = useState<SomeType | null>(null)
   const end = useRef(false)
   const handleResult = useCallback((res)=>{
      // 要么全部一次变更，要么全部不变更
      if(end.current) return
      setResult(res)
      // 其他操作，一次性解决
      // setXxx()
   },[])
   const submit = useCallback(()=>{
       request().then(handleResult)
   },[])
   useEffect(()=>{
      return ()=>{
        end.current = true
      }
   },[])
   return <button onClick={submit}/>
}

这种实现有一种专有名词 —— 最终一致性，即不关心事务产生中途的脏数据，只保证事务结束时的特性即可

这样的多个 useCallback，形成了一种类似管道的结构，这是类似 Rx 的结构（当然 rx 的功能和实现要复杂的多）

你会发现，所有问题集中在了 useEffect 这个 api 上，关键在于能不能不使用 useEffect ？

useEffect 的作用，是在状态变化后，等到视图也进行了响应，执行相关逻辑，属于状态 + 视图 的响应式

而看了之前文章的朋友们，会发现在 ng 当中，没有直接方法可以让你做出单纯状态的响应式，但是你可以自己实现：

@Component({/* ... */})
class SomeCompo{
  _data: string = ''
  get data(){
     return this._data
  }
  set data(res:string){
     this._data = res
  }
}

因为这个逻辑是你自己实现的，与组件逻辑没有耦合，因此，当组件销毁之后，set data 的逻辑还可以继续执行

vigilant-smoke-wjuzm - CodeSandbox

而 React 类似的逻辑，直接在 useEffect 处就被打断：

young-waterfall-rpjjc - CodeSandbox

因为 ng 组件 的逻辑代码与视图渲染没有任何关系，与视图耦合的只有 Input，Output，ViewChild 装饰器以及一些生命周期，基本上我们在组件开发的时候，用不到生命周期，除了精确地获取节点（非精确一个 setTimeout 0 即可），处理 SSR 以外，你可以翻阅 Mat 或者 CDK 源码，会发现用到生命周期的地方非常少

所以当你针对 Input Output 以及管道进行开发时，它就是和 React 一模一样的编程模型，前提是你关掉 onPush

当组件被销毁时，因为存在相关属性和方法的引用，按引用计数，依然会进行相关调用

所以，从事务角度分析，ng 不需要状态管理

同理，作为对异步和事务流程的建模，rx 本身并不适合 React，因为一旦你使用了 Rx，就会出现大量的 setState 报错，比如：

const bs = useRef(useBehaviorSubject(''))
const bsDebounced = useRef(bs.current.pipe(debounceTime(1000)))
const bsState = useObservable(bs)

// useObservable 实际上就是一个 useState + subscribe 时 setState + 组件销毁时 unSubscribe
function useObservable<T>(ob: MutableRefObject<T>){
   const [state, setState] = useState<T>()
   useEffect(()=>{
      const subscription = ob.subscribe(res=>{
          // 问题来了，组件销毁时，setState 是会报错的
          setState(res)
      })
      return ()=>{
          subscription.unsubscribe()
      }
   },[ob])
}

// bsState 很难 useEffect 使用
useEffect(()=>{
   // 组件销毁中进行的流操作，不会引发 useEffect
   // 破坏事务原子性
},[bsState])

那好，你说，我不写 useEffect，全程 useRef + useObservable，单纯只用 Observable 的响应式，可不可以？

不行，因为 useEffect 在父子组件间传值时非常重要

function Compo(props:{val:string}){
   const [local, setLocal] = useState('')
   useEffect(()=>{
      setLocal(props.val)
   },[props.val])
}

// 而不像 ng 中，子组件 local 可以直接代理
@Component({})
class SomeCompo{
   _data = ''
   @Input()
   get data(val){
      return this._data
   }
   set data(val: string){
      this._data = val
   }
}

所以，你办不到只用 rx，不用 useEffect

况且，就算你能够办到，你确定满屏幕 useRef 的代码，是 React 代码么？是否是技术选型出现了问题？与第三方库又如何对接？会不会 Cycle.js 搭配 jsx 会更好？

所以一般意义上认为，Rx 搭配事件驱动系统（比如 ng 或者 android），会比搭配 React 更好，这也算是解答之前某位热心读者的问题
