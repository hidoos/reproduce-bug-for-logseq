react hooks 的 useReducer 有什么用？

这个问题被讨论过很多遍，但是在 React 平台，很多使用者只是单纯以为 useReducer 是 redux 的简化版

但是为什么用 redux，这个问题很多人没有理解，实际上从效果上来看，useReducer 和 redux 是完全不同的

reducer 本质是数据行为分离

为什么函数式的实体构建一定要涉及数据行为分离呢？因为函数式中没有变量，因为没有变量，所以你获取到的每个 cb，都是次次不同的，cb 与普通状态本质一样

假设状态的变化会导致副作用执行，那么 cb 也会如此，因此 ——

必须保证不可变且唯一的 cb —— dispatch

也因为如此，React 中的 useReducer，useState，以及各种自己实现的 Hooks，一旦涉及 IO，必须全部封装成 readonly [state, dispatch] 的结构

这样的话，可以：

const [state, dispatch] = useContext(Service)

useEffect(()=>{
   // dispatch 传入依赖，不会导致 effect 多执行
   // 外部函数和 useCallback(()=>{},[]) 也如此
},[dispatch])

return <div>{
   useMemo(()=><Compo>
     {/*  useMemo 避开 state 即可防止服务影响子组件  */}
   </Compo>,[])
}</div>

// 分开注入 state 和 dispatch，可以保证子孙组件 FC
// 仅仅在 state 变化时重新运行
<ServiceState.Provider value={state}/>
<ServiceDispatch.Provider value={dispatch}/>

这种做法，你只要页面涉及倒计时，又不想或无法用组件封装倒计时逻辑，就会碰到，并亲自体验一番

换句话说，只要明白这个逻辑，不难发现，useReducer 只是对 useState 的框架级别封装而已：

const [state, setState] = useState({name: '', password: ''})
const dispatch = useCallback((action: {name:string, payload:any})=>{
   setState(res=>{
       if(action.name==='changeName'){
            return {...res, name: action.payload}
       }
       return res
   })
},[])
return [state, dispatch] as const

目的就是让 useState 封装行为，仅此而已

翻译成 class 风格，useState 即：

class Some{
   _data = ''
   set data(res:string){
      this._data = res
   }
   get data(){
      return this._data
   }
}

而 useReducer 相当于：

class Some{
   _data = ''
   set data(res:string){
      this._data = res
   }
   get data(){
      return this._data
   }
   changeName(payload:string){
      this.data = payload
   }
}

你可能会说，数据行为分离怎么比 class 难写那么多，为什么要这么用？

几个原因：

  1. 不变性使得数据可以判等，框架可以更方便地进行处理，以及实现响应式
  2. state 可以很方便地进行序列化和消息传递（适合前端场景）
  3. 手动声明初始值，方便进行单元测试

这其实就是 DDD 倡导值对象的原因，值对象因为其不变性，对于消息传递和持久化非常友好，即 —— 全部用值对象比全部用实体要好！

不过，还有一个负面的情况，就是 JS/TS 的 useReducer 写起来过于麻烦，且 useReducer 或者说数据行为分离，压根不是为 JS/TS 准备的：

模式识别，校验类型，不变性是函数式的特点，同时也是实现状态行为分离的基础设施

  • 模式识别避免冗长的 if 语句
  • 不变性防止麻烦的对象变更
  • 校验类型避免恶心的 action 声明

真实开发环境中，工程都是需要考虑成本和收益的，因为生态和工具链问题，很少有团队会选择 ocml 或者 rescript，因此没有以上特性支持

而在没有以上特性支持的情况下，用 useReducer 甚至 Redux 就是个投入产出不成正比的行为。你将代码写得如此冗长，没有特性支持的情况下出现大量样板代码，接入其他 hooks 也会变得困难，实际仅仅是防止了 cb 变更，以及未分开注入导致的深层组件 FC 多跑几次，多几次 Diff

神经正常的人知道怎么选择，cb 变更可以通过 useRef 消除依赖，使得依赖清空为 [], 而 FC 可以对组件节点针对性 useMemo 解决

虽然后者不符合 React 编程思想，但是它有效，且快速

当然，如果实在对象过于复杂，且有一定的事件准确性要求，上 useReducer 也非常合理，这就是 React 官网说的，复杂嵌套对象和事件的情况

当然，这种情况，属于极个别情况，应用当中不会出现太多的，不影响 React 已经宣判 Redux 等一众状态管理库死刑的现状
