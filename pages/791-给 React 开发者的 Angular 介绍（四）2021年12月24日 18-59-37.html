今天我们来重点关注前端框架的另一个内容： 样式封装

当我们将视图封装成组件，逻辑封装成服务（自定义 Hooks）时，会忽略非常重要的一点 —— 样式

前端中的组件指 UI 封装单位（来自 React 文档定义），逻辑封装袋为服务（SOA，或 React hooks）

样式的复用在原生代码中，是与 js 代码隔离开的，不过可以通过 DOM 进行访问和变更

React 的想法是 css in js，即样式也作为代码的一部分，被封装于组件之中

常见的方案如 styled-components，也是个人用的最多的方案，的确非常方便

但是这种方案留下了一堆 div （毕竟常用 styled.div），在严苛的 HTML 语义化标准下，并不符合要求

当然，如同 a11y ，焦点管理，浮动层管理，标准多语言文件 .xlf 这类需求，React 或 Vue 平台默认忽略掉一样，语义化要求下的样式封装也被忽略掉了

所以，当你想要开发严谨应用时，这类方案需要斟酌，但是可以肯定的是，ng 在这方面会更加正规，并正视很多大部分需求下被认为是可以舍弃不谈的问题

W3C 标准

React 或 Vue 应用，默认生成的组件，是直接将组件整体作为 virtual dom 插入，这并不符合 W3C 的 webComponent 标准，甚至不符合（HTML import，HTML template，shadow dom）三者中任何一个

不符合标准的问题，即标准的配套生态无法准确识别你的内容，这对于屏幕阅读器，对于一些 a11y 内容，对于搜索引擎，甚至对于内容解析软件，都是非常不好的影响

而在 ng 中，组件选择器直写内容时，即采用 Custom Elements 标准：

@Component({
  selector: "app-emulate",
  encapsulation: ViewEncapsulation.Emulated,
})
export class AppEmulate

看到我们提供了一个属性，这个 encapsulation，默认值为 emulated，还有 none 和 shadowDom

当其为 none 时，为全局样式，而当期为 emulated 时，样式配置会加上组件绑定的属性名（_nghost-jec-c1）

这部分逻辑与 Vue 相同，但是 Vue 至少目前没有 :host 选择器，而 Vue 没有 host 选择器的很大部分原因，是因为没有遵循 custom elements 的 web component 标准

带来的结果，便是 Vue 文档中的一段解释：

这样会代码很严重的问题，大家使用 Vue 的时候，经常习惯性地给跟元素加上 之类的类名，然而根节点的类名，不具备封装性，相信使用过 Vue 的朋友都被这个逻辑坑害过

而 ng 则不同：

  1. 有进有出（全局） None 配置
  2. 有进不出 （仿真） Emulated 配置，:host 配置根节点
  3. 不进不出（shadowDom） ShadowDom 配置， 此时 ng 的 全局样式和 None 配置生效，但是引入的 css 包（<link>）不生效，:host 同样可以配置根节点

后果就是，单纯追求封装性的话，Vue 本身不合格（根节点），需要搭配 css Module，而 css Module 在 Vue 中的使用体验，真的谈不上友好

追求封装性的话，React 非常优秀，尤其是很多 css in js 库，但是谈论 “样式封装” 这个话题，则不合格，因为整个前端应用的各个部分，牵一发而动全身，既然 W3C 已经定下了标准，这就不是某个厂家甚至某个框架的事情，而是大家都应该以此为目标，shadow Dom 不支持也罢，但是 Custom Elements 应该遵守吧？

而 ng，遵循全部 W3C 标准

所以当你需要严谨地去实现一个前端视图时，ng 会带来更好的支持

况且，这部分不仅仅是视图的事情，且不说前面提到 a11y 等需求，当你涉及到多应用内容时 —— 比如不使用 iframe 封装前端应用 —— 那么样式封装就会变成非常复杂的问题，你也不会遇到 React 会遇到的，只有引入不引入父应用样式这样的单选题，相信有过尝试的朋友都有体会

而对于 ng，开发单个应用的样式封装，和开发多个应用的样式封装，都是同一个逻辑（甚至应用就是 Module，原生支持领域封装）

这也是 ng 设计的基础，为了同时支持 CustomElement，shadow Dom，ng 就不得不抛弃 React/Vue 类似的组件构建方式，转而采用继承指令的方式（为已有视图附加逻辑），这之后的一切 ng 特性，说不定都由此而来
