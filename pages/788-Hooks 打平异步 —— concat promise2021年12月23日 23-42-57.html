小朋友们在利用 promise 或者 aysnc await 语法糖处理异步的时候，经常遇到这样的问题：

// 一个异步函数
const request = useCallback(async ()=>{},[])

// 连续多次调用

遇到请求等通讯时间不确定的异步操作时，容易出现冲突

比如发起顺序 requestA -> requestB -> requestC

结果回调却可能是 requestB -> requestC -> requestA

你可能会通过业务解决此类问题

比如 request() 发起时，取消上次请求（与 promise 语义冲突），又或者通过 debounce 使其在大部分情况下运行正常

今天来看看另外一个解法：

即，按照你发起时的顺序，后序异步总在前者返回之后进行

这个逻辑该如何实现呢？

其实非常简单，我们只需要构造一个 WeakMap，它拥有如下结构：

interface AsyncAction<P extends any[], R> {
    params: P,
    handler: (...args:P)=>Promise<R>
}

const actionPool:WeakMap<AsyncAction<P,R>, AsyncAction<P,R>|undefined>(new WeakMap)
为什么要用 WeakMap ？ 纯粹不想处理垃圾回收问题，因为 WeakMap 是弱引用，如果目标对象销毁，则关联关系销毁

这个 actionPool，则标识后一 action 在 前一 action 之后运行：

由此封装了一个 useAsync 函数，使用非常简单：

export default function App() {
  const time = useRef(Date.now());
  const { trigger, loading } = useAsync(
    wait,
    // onSuccess
    () => {
      // 打印一下时间
      const now = Date.now();
      console.log("period finish, duration:" + (now - time.current) + "ms");
      time.current = now;
    },
    // onError
    undefined,
    true,
    // onSequenceSuccess
    () => {
      console.log("sequence finish");
    }
  );
  return (
    <div className="App">
      <button
        onClick={() => {
          if (!loading) time.current = Date.now();
          trigger(1000);
        }}
      >
        click
      </button>
      <div>is loading? {loading + ""}</div>
    </div>
  );
}

嗯呐，这么说没错，但是 FC 重新运行代表重新渲染么？ FC = class 的 this.render 么？

不是

首先：因为带 Hooks 的 FC 中，Hooks api 有 memo 功能，函数多次运行并不代表每次都会运行 Hooks 功能

其次：你可以通过 useMemo 控制每一个单一节点的渲染

这也就是 React 再解释 memorize 优化时提到的：

用 useMemo 优化每一个具体的子节点

这其实不叫优化，因为你没有将视图看做一个单一层级，而是 Hooks 的延伸，Hooks 确定了调度，那么 ReactNode 也应该跟，就这么简单

你也可以不优化，因为你知道自己在干什么，你有每一个节点的渲染控制权

有每一个节点的渲染控制权，这个概念很重要，因为 Class api 中，你无法单独控制 render 中的某一个节点的渲染，你只能将其作为一个整体进行渲染，但是 FC with Hooks 可以
当渲染对于你来说是黑盒时，你需要利用它提供的性能优化 api 来调优，但是 FC with Hooks 下，渲染调度不是黑盒，完全可控，责任在你

也因此，我们不考虑会有什么视图影响，我们知道我们能完全控制节点渲染，因为实现功能时忽略此事，那么，useAsync 应该如此封装：

function useAsync<P extends any[], R>(params: P, cb: (...args: P)=> Promise<R>){
   // ... 逻辑和其余参数同上一版本
   useEffect(()=>{
       cb(...params).then()
   },[params, cb])
}

即，params 和 cb 变化时，就直接运行异步，且加上 concat 之类的逻辑

这样，你也能够明白，为什么 React 18 中会可能存在 useDeferedValue 之类的原生 API 了

当然，这里也能解释，那些地方要用 useCallback，而哪些地方不用

你跨组件，回调需要被用作调度指示的时候，必须用 useCallback，而类似合成事件出，则不需要，因为回调只可能在当前组件，且不会被 useEffect 监听，仅从 useCallback 拿的是快照值这个点上，并不能提供指导意义

因为 useCallback， 其实就是 Action，众多 CSP 形式中的一种而已

对了，如果你了解 Rx，你会发现 Rx 也是如此，比如 SchedulerLike 和一堆 Action 结构，不过 Rx 会进一步封装，将异步执行监听后的数据与异步也建立联系，即 Actor 模型

之前有讲过 构建实体对象方案 和 全部值对象（状态管理） 方案的区别，感觉很多人没看清，以为还有其他路线走，其实并没有，就这两条路线，个人实践认为 React Hooks 更适合前一种，而 Class api 更适合状态管理，他们也分别对应建模优先和视图优先两种风格

看你的应用场景，进而选择适合自己的方案，而且两者并非水火不容
